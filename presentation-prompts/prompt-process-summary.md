# 프롬프트 설계 과정 정리

이 문서는 중고 매물 분석 도구의 발표 자료를 만들기 위해, AI 호출 과정 중 "유사 매물 검색 키워드 생성"과 "제품 고질병·관련 이슈 검색" 프롬프트를 어떤 문제의식으로 설계하고 개선했는지 정리한 발표용 메모다. 추후 HTML 기반 5페이지 슬라이드로 옮길 수 있도록 흐름 중심으로 작성했다.

## 0. 전체 AI 처리 흐름

프로젝트 전체에서 AI 호출은 한 번으로 끝나는 구조가 아니다. 확장프로그램이 매물 정보를 웹 분석 화면으로 전달한 뒤, 목적이 다른 AI 호출들이 순서대로 이어지는 구조다. 현재 문서에서 자세히 설명하는 검색어 프롬프트는 그중 1단계에 해당한다.

1. 제품 식별·키워드 추출·관련 시세 검색
   - 확장프로그램이 번개장터·당근마켓 매물 상세 페이지에서 제목, 본문, 가격, 판매자 정보, 이미지 URL을 수집해 분석 웹으로 보낸다.
   - AI가 사진·제목·본문을 보고 이 매물이 어떤 제품인지 식별한다.
   - 식별된 제품을 기반으로 동일·유사 매물을 찾기 위한 검색 키워드를 만든다.
   - 그 키워드로 관련 시세 매물을 수집한다.
   - 단, 부속품·세트 구성·구성품 누락 여부가 섞이면 시세 신뢰도가 떨어질 수 있으므로, 이 문제는 이후 별도 보정 단계로 다룰 예정이다.

2. 제품 고질병·관련 이슈 검색 및 사진 기반 하자 탐지
   - 식별된 제품명을 기반으로 해당 제품의 고질병, 자주 발생하는 하자, 구매 전 확인해야 할 이슈를 검색한다.
   - 판매자가 올린 사진을 AI가 분석해 찍힘, 기스, 파손, 구성품 누락 의심 등 시각적 하자를 찾는다.
   - 향후에는 찾아낸 하자를 좌표 기반으로 서버에 전달하고, 서버가 이미지 위에 직접 오버레이를 그려 표시하는 기능을 고려한다.
   - 이 단계는 단순 텍스트 요약이 아니라, 제품 지식과 이미지 비전 분석이 결합되는 단계다.

3. 시세·하자·판매자 정보 기반 가격 판단
   - 1단계에서 수집한 관련 시세와 2단계에서 찾은 제품 이슈·사진 하자·판매글 정보를 함께 사용한다.
   - AI가 이 매물이 적정가인지, 비싼지, 싼지, 사기 의심 정황이 있는지 판단한다.
   - 최종적으로 합당한 추천 가격 또는 가격 범위를 제시한다.
   - 이 단계는 단순 평균가 계산이 아니라, 상태와 리스크를 반영한 가격 판단 프롬프트가 필요하다.

4. 판매자에게 보낼 네고 대사 생성
   - 가격 판단 결과와 발견된 하자·시세 근거를 바탕으로 판매자에게 보낼 협상 문장을 만든다.
   - 무례하지 않으면서도 근거가 있는 네고 문구를 생성하는 것이 목표다.
   - 예를 들어 "시세보다 조금 높고, 사진상 사용감이 보여서 이 가격에 가능할까요?"처럼 사용자가 바로 보낼 수 있는 형태가 되어야 한다.

따라서 발표에서는 "AI가 한 번에 모든 판단을 한다"가 아니라, 제품 식별, 하자·이슈 분석, 가격 판단, 네고 문장 생성처럼 목적이 다른 AI 호출을 단계적으로 연결한 구조라고 설명하는 것이 적절하다.

## 1. 목표

이 문서에서 집중하는 기능의 목표는 사용자가 중고 플랫폼의 매물 상세 페이지를 열었을 때, AI가 사진·제목·본문을 함께 보고 같은 상품을 찾기 위한 가장 적절한 검색 키워드를 자동으로 도출하는 것이다.

단순히 판매자가 쓴 제목을 복사하는 것이 아니라, 실제 번개장터·당근마켓 검색창에 넣었을 때 동일하거나 유사한 매물이 잘 나오는 제품명·모델명·통칭을 만드는 것이 핵심이다.

## 2. 초기 문제

초기에는 AI가 판매글의 표현을 그대로 따라가거나, 검색어를 너무 짧게 잘라내는 문제가 있었다.

- `닌텐도 스위치2`가 `닌텐도 스`처럼 중간에서 잘림
- `반스 어센틱`이 `어센`처럼 불완전하게 줄어듦
- `새상품 같은 가방찾는다면~`, `단순개봉 플레이 판매`처럼 판매자의 홍보·상태 설명이 검색어에 섞임
- 후보 출력 과정에서 `json`, 코드펜스, 설명 문장이 버튼으로 노출됨
- 프롬프트가 특정 예시를 직접 들면, 모델이 그 예시를 일반 규칙처럼 오해할 위험이 생김

이 문제들은 모두 "제품 식별어"와 "판매글 맥락"을 구분하지 못할 때 발생했다.

## 3. 설계 방향

프롬프트는 특정 상품명이나 특정 금지어를 계속 추가하는 방식으로 만들지 않았다. 발표에서 강조할 핵심은 "하드코딩된 제품별 규칙이 아니라, 모든 매물에 적용되는 판단 절차"다.

현재 프롬프트의 중심 원칙은 다음과 같다.

- 사진, 제목, 본문을 함께 본다.
- 판매자 표기가 틀릴 수 있으므로 사진과 웹 검색으로 확인한 사실을 우선한다.
- 출력 전 내부적으로 단어를 두 부류로 나눈다.
- 제품 식별어: 브랜드, 라인, 모델, 세대, 품목
- 판매글 맥락: 상태, 거래조건, 사용 이력, 배송, 인증 가능 여부, 홍보 설명
- 최종 검색어에는 제품 식별어만 남긴다.

예를 들어 `닌텐도 스위치2 단순개봉 플레이 판매`는 판매 상태와 판매 행위를 제거하고 `닌텐도 스위치2`처럼 남겨야 한다. `지방시 안티고나 핸드백 (새상품 같은 가방찾는다면~)`은 권유 문장을 제거하고 `지방시 안티고나 핸드백`처럼 남겨야 한다.

## 4. 프롬프트 파일 분리

교수님께 프롬프트 과정을 보여주기 쉽도록 AI에게 전달하는 프롬프트는 `prompts` 폴더로 분리했다.

- `prompts/search-query-single.txt`: 최종 자동 수집에 사용할 단일 검색어 생성 프롬프트
- `prompts/search-query-candidates.txt`: 개발·검수 과정에서 최대 3개의 후보를 비교하기 위한 프롬프트

두 프롬프트는 같은 철학을 공유한다. 후보 생성은 사용자가 최종적으로 고르기 위한 기능이 아니라, 품질 확인과 자동 선택 로직 개선을 위한 중간 장치다. 최종 목표는 AI가 가장 적합한 키워드를 자동으로 선택하고, 그 키워드로 유사 매물을 수집하는 것이다.

## 5. 후처리 보강

프롬프트만으로는 모델 출력이 항상 안정적이지 않기 때문에 서버에서도 후처리를 적용했다.

- 코드펜스, JSON 키, 설명 문장 제거
- 단어 중간에서 잘린 검색어 보정
- 제목 전체를 그대로 복사하지 않도록 제한
- 상태·거래·사용 설명이 시작되는 지점 이후 제거
- 후보가 너무 길거나 판매글 문장처럼 보이면 버림

이 후처리는 프롬프트를 대체하는 것이 아니라, 모델 출력의 안전장치 역할을 한다.

## 6. 속도 최적화

정확도를 위해 이미지와 Google Search grounding을 사용했지만, 모든 요청에서 원본 이미지와 웹 검색을 무조건 사용하면 응답이 1분 가까이 느려질 수 있었다.

현재 구조는 다음처럼 최적화했다.

- 이미지는 최대 3장만 사용한다.
- 이미지 다운로드는 병렬로 처리한다.
- 이미지 1장당 타임아웃을 6초로 제한한다.
- 가능한 경우 400px급 이미지 URL로 낮춰서 전송한다.
- 먼저 빠른 멀티모달 응답을 시도한다.
- 결과가 부적절할 때만 Google Search grounding으로 재시도한다.

이렇게 해서 일반적인 매물은 빠르게 처리하고, 식별이 어려운 매물만 웹 검색을 사용하도록 했다.

## 7. 발표 슬라이드 초안

나중에 HTML 슬라이드로 만들 때는 다음 5페이지 구성이 적합하다.

1. 문제 정의: 중고 매물은 제목·사진·상태·시세가 불완전해 사람이 직접 판단하기 어렵다.
2. 전체 AI 파이프라인: 키워드·시세 검색, 하자·이슈 분석, 가격 판단, 네고 대사 생성.
3. 1단계 상세: 제품 식별어만 남기는 검색어 프롬프트 설계와 유사 매물 수집.
4. 확장 예정 단계: 제품 고질병 검색, 사진 하자 탐지(홍보 이미지 vs 실물 사진 구분), 좌표 기반 이미지 오버레이.
5. 최종 가치: 적정 가격 추천과 근거 있는 네고 문장까지 이어지는 구매 의사결정 보조.

## 8. 현재 결론

현재 프롬프트의 핵심은 "AI에게 특정 상품별 정답을 외우게 하는 것"이 아니라, 어떤 매물이 들어와도 제품 식별에 필요한 단어만 남기는 판단 절차를 주는 것이다. 이 방향이 발표에서도 가장 설득력 있는 설명 포인트다.

## 9. 제품 요약 프롬프트로 확장

검색어 생성만으로는 사용자가 "이 매물이 정확히 어떤 제품인지", "신품가는 어느 정도인지", "국내에서 누가 판매하는지"를 바로 판단하기 어렵다. 그래서 AI 처리 흐름을 단일 검색어 생성에서 제품 요약 단계까지 확장했다.

현재 제품 요약은 한 번의 AI 호출에 모든 것을 맡기지 않는다. 역할을 나눠 두 단계로 처리한다.

1. `prompts/product-identify.txt`
   - 판매글 제목, 본문, 사진을 보고 실제 제품명을 먼저 식별한다.
   - 신품가, 제조사, 설명을 쓰지 않고 제품 식별에만 집중한다.
   - 판매자의 상태·거래 조건·가격·홍보 문구는 제거한다.

2. `prompts/product-summary.txt` 및 서버 내부 제품 정보 조회 프롬프트
   - 1단계에서 식별한 제품명으로 Google 검색을 수행한다.
   - 한국 기준 신품가 또는 국내 정가/가격대, 제품 설명, 제조사·퍼블리셔·국내 판매처를 정리한다.
   - 같은 제품을 찾기 위한 검색어 후보도 함께 만든다.

이 구조는 "판매글을 요약하는 AI"가 아니라 "판매글에서 제품을 식별한 뒤, 제품 자체 정보를 다시 검색하는 AI"라는 점이 중요하다. 판매자의 설명이 부정확하거나 짧아도, 식별된 제품명으로 다시 검색하기 때문에 제품 정보의 안정성이 높아진다.

## 10. 검색어 품질 개선

실제 테스트에서 검색어에 `중고`, `가격`, 영문명 단독 후보가 섞이는 문제가 있었다. 예를 들어 `동키콩 바난자 가격`이나 `Donkey Kong Bananza Nintendo Switch 2`처럼 나오면, 번개·당근 검색어로는 너무 넓거나 한국 중고 플랫폼 검색 관점에서 어색하다.

이를 해결하기 위해 프롬프트와 서버 후처리를 함께 보강했다.

- 검색어에는 `중고`, `가격`, 상태, 지역, 거래 조건을 넣지 않도록 명시했다.
- 검색어는 한국어 표기를 우선하고, 영문명만 단독으로 쓰지 않도록 했다.
- 서버의 정규화 단계에서도 `중고`, `가격`을 제거하고, 한글이 없는 후보는 버린다.
- 여러 검색어가 한 줄에 섞여 나오면 쉼표·슬래시·줄바꿈 기준으로 분리해 각각의 검색 칩으로 보여준다.

발표에서는 이 부분을 "프롬프트 규칙만 믿지 않고, 서버 후처리로 한 번 더 안전장치를 둔 사례"로 설명할 수 있다.

## 11. 제품 이미지 검색 개선

초기 제품 이미지 기능은 검색 결과 첫 이미지를 그대로 보여주다 보니 이미지가 너무 작거나, 깨진 이미지가 뜨거나, 같은 프랜차이즈의 다른 시리즈 이미지가 뜨는 문제가 있었다.

개선 방향은 다음과 같다.

- 제품명과 검색어에서 핵심 단어를 뽑아 이미지 검색 후보를 점수화한다.
- 후보 이미지가 실제로 로드 가능한지 서버에서 먼저 검증한다.
- 외부 이미지 URL을 그대로 화면에 넣지 않고, 로컬 `/api/image-proxy`를 통해 보여준다.
- 분석 화면에서는 제품 이미지를 작은 썸네일이 아니라 대표 이미지 영역으로 크게 보여준다.
- 이미지 갱신은 임시 성격의 기능이므로 큰 버튼이 아니라 작은 새로고침 아이콘으로 낮춰 표시한다.

이 단계는 AI가 제품 정보를 찾는 것과 별도로, 사용자가 제품을 시각적으로 확인할 수 있게 돕는 보조 장치다.

## 12. 분석 화면 UX 피드백 반영

프롬프트 결과가 좋아도 화면에서 읽기 어렵거나 버튼 의미가 모호하면 사용성이 떨어진다. 그래서 실제 테스트 화면을 보면서 다음 UI 조정을 반복했다.

- 확장 버튼과 플로팅 버튼은 같은 동작으로 통일했다.
- 관련 매물은 페이지 로드 시 자동으로 뜨지 않고, 사용자가 버튼을 누를 때만 수집한다.
- `관련 매물 분석하기`는 의미가 딱딱해서 `비슷한 매물 찾기`로 바꿨다.
- `원본` 링크는 `판매글 열기`로 바꿔 클릭 후 이동할 위치를 더 명확히 했다.
- 본문·제품 설명은 말줄임만 하지 않고, 길 때만 내부 스크롤 영역을 만든다.
- 짧은 텍스트에는 스크롤 영역과 박스 스타일을 만들지 않아 카드 안에서 자연스럽게 보이도록 했다.
- 카드 안 요소가 테두리에 붙어 보이지 않도록 공통 패딩과 제품 요약 카드의 2열 레이아웃을 정리했다.
- 매물 사진 카드 제목은 `사진 3` 대신 `매물 사진`으로 바꿔 더 자연스럽게 했다.

발표에서는 이 과정을 "AI 프롬프트 설계와 화면 피드백이 같이 개선된 사례"로 정리할 수 있다. 모델 출력이 맞아도 사용자가 읽고 행동하기 쉬운 화면 구조가 필요하다는 점을 보여준다.

## 13. 현재 시연 흐름

현재 시연에서 보여줄 수 있는 흐름은 다음과 같다.

1. 사용자가 번개장터 또는 당근마켓 매물 상세 페이지에서 확장 버튼을 누른다.
2. 확장 프로그램이 제목, 본문, 가격, 판매자 정보, 사진 URL을 분석 웹으로 보낸다.
3. 분석 웹은 먼저 매물 기본 정보와 사진을 보여준다.
4. AI가 제품명을 식별하고, 식별된 제품명으로 제품 정보를 다시 검색한다.
5. 제품 요약 카드에 제품명, 국내 기준 신품가, 판매처/제조사, 제품 설명, 검색어 후보, 대표 이미지가 표시된다.
6. 사용자가 `비슷한 매물 찾기` 또는 검색어 칩을 누르면 번개·당근 검색 탭을 열고 비교 매물을 수집한다.

이 흐름은 "중고 매물 하나를 사람이 직접 검색하고 비교하는 과정"을 AI와 확장 프로그램이 단계별로 자동화한 것이다.

## 14. 제품 고질병·관련 이슈 프롬프트 시행착오

2단계 기능의 목표는 식별된 제품명을 바탕으로, 중고 구매자가 사기 전에 반드시 확인해야 할 제품 고질병과 관련 이슈를 카드 형태로 보여주는 것이다. 테스트 제품은 `Steam Deck OLED 512GB` 매물을 사용했다.

처음에는 제품 리스크 프롬프트를 서버 코드 안에 직접 문자열로 작성했다. 이후 발표와 유지보수를 위해 `prompts/product-risk.txt`로 분리했다.

초기 프롬프트의 핵심 구조는 다음과 같았다.

```text
중고 구매자가 나쁜 매물을 피할 수 있도록 "{{productName}}"의 제품 리스크를 Google 검색으로 직접 조사해 정리하세요.

제품명 그대로와 "{{productName}} 문제", "{{productName}} 고장", "{{productName}} defect", "{{productName}} issue", "{{productName}} 중고 주의"처럼 실제 웹에서 쓰이는 표현을 스스로 찾아보세요.

반드시 JSON만 출력하세요:
{
  "relatedIssues": [
    { "title": "검색된 실제 이슈명", "detail": "왜 중고 구매자가 신경 써야 하는지 구체적으로 설명", "level": "safe|caution|risk" }
  ],
  "chronicDefects": [
    { "title": "구체적 증상/부품명", "detail": "구매 전 확인해야 하는 이유와 실제로 어떤 문제가 생기는지 설명", "level": "safe|caution|risk" }
  ]
}
```

이 방식의 문제는 "조사"와 "JSON 형식 준수"를 한 번에 강하게 요구했다는 점이다. 특히 `근거가 약하면 배열에 억지로 넣지 마세요`, `확실한 고질병이 적으면 빈 배열을 반환하세요` 같은 규칙이 함께 들어가면서, 모델이 실제로는 답할 수 있는 제품도 보수적으로 비워 버리는 현상이 생겼다.

실제 자동 2단계 출력은 다음처럼 나왔다.

```text
관련 이슈
스팀 덱 OLED 512GB 웹 이슈 부족
검색에서 뚜렷한 제품 이슈를 찾지 못했습니다. 2차 확인이 필요합니다.

고질병 조사
스팀 덱 OLED 512GB 고질병 정보 부족
검색에서 반복 고장 정보를 충분히 찾지 못했습니다.
```

토큰 제한이 문제일 가능성을 확인하기 위해 `maxOutputTokens: 2800`도 제거했다. 그러나 결과는 크게 달라지지 않았다. 따라서 단순히 출력 길이 문제가 아니라, 프롬프트 구조 자체가 모델의 검색·추론보다 JSON 안전성을 우선하게 만든 것이 원인이라고 판단했다.

이를 검증하기 위해 상단에 `AI 대화` 창을 만들었다. 이 창은 사용자가 입력한 문장을 그대로 `/api/ai-chat`으로 보내고, 같은 Gemini 모델에 Google Search를 켠 상태로 자유 응답을 받는 디버그용 대화창이다.

직접 대화창에서 다음처럼 물었다.

```text
Steam Deck OLED 512GB 중고매물을 사려는데 고질병이나 관련 이슈를 알려줘.
```

이때 모델은 정상적으로 자세한 답변을 생성했다. 출력에는 다음 항목들이 포함됐다.

- OLED 패널 번인 가능성
- 데드 픽셀 또는 화면 얼룩·mura 확인
- 버튼·트리거·스틱 입력 문제
- 오디오 노이즈 가능성
- 배터리 효율·헬스 확인
- SSD 개조 여부와 교체 흔적
- Wi-Fi 또는 블루투스 간섭
- RMA·보증·영수증 양도 여부

즉 모델 자체가 정보를 못 찾는 것이 아니라, 자동 2단계 프롬프트가 너무 형식 중심이라 결과를 비워 버린다는 점이 확인됐다.

이후 2단계 파이프라인을 두 번의 AI 호출로 나눴다.

1. `prompts/product-risk.txt`
   - Google Search를 사용해 제품 고질병과 관련 이슈를 자유롭게 조사한다.
   - JSON을 요구하지 않고 자연어로 충분히 자세히 설명하게 한다.

2. `prompts/product-risk-json.txt`
   - 1차 조사 메모를 입력으로 받아 UI 카드에 넣을 JSON만 만든다.
   - 이 단계는 검색을 다시 시키지 않고, 조사 결과를 구조화하는 역할만 한다.

최종 `product-risk.txt`는 “직접 대화에서 사용자가 물어본 것처럼” 제품명을 전제로 중고 구매 리스크만 묻는 형태로 정리했다. 중간에 “현재 날짜 기준 최신성 규칙”이나 “제품 실존 여부·공식성 검증”을 넣어 보았지만, 오히려 모델이 `공식 제품 불확실`, `비공식/사기 가능성` 같은 방향으로 새는 문제가 생겼다. 따라서 2단계는 앞 단계에서 이미 식별된 제품을 신뢰하고, 제품 진위 검증이 아니라 실제 구매 체크 포인트만 조사하도록 고정했다.

```text
당신은 중고 거래 분석가입니다.

아래 제품을 중고로 사려는 사람에게, 실제로 확인할 가치가 있는 관련 이슈와 고질병/반복 리스크를 조사해 설명하세요.
답변은 직접 대화에서 사용자가 "{{productName}} 중고매물을 사려는데 고질병이나 관련 이슈를 알려줘."라고 물었을 때처럼 자연어로 작성하고, JSON으로 쓰지 마세요.

절차:
1. 제품 요약 정보는 이미 앞 단계에서 식별된 결과입니다. 이 단계에서는 제품의 실존 여부나 공식성 검증을 다시 하지 말고, 중고 구매 리스크만 조사합니다.
2. Google 검색으로 "{{productName}} 문제", "{{productName}} 이슈", "{{productName}} 후기", "{{productName}} 불량", "{{productName}} AS", "{{productName}} 주의"처럼 필요한 쿼리를 조합해 실제 근거를 찾습니다.
```

그리고 `product-risk-json.txt`는 조사 메모를 받아 다음 형식으로만 변환한다.

```text
아래 제품 리스크 조사 메모를 중고 구매자용 카드 JSON으로 정리하세요.

제품명: {{productName}}

조사 메모:
{{researchText}}

반드시 JSON 객체 하나만 출력합니다.
```

서버 코드도 다음처럼 바뀌었다.

```js
const researchText = await geminiGenerateFromParts(apiKey, model, [{ text: researchPrompt }], {
  useGoogleSearch: true,
  temperature: 0.35,
});

const jsonText = await geminiGenerateFromParts(apiKey, model, [{ text: jsonPrompt }], {
  responseMimeType: 'application/json',
  temperature: 0.05,
});
```

수정 후 자동 2단계에서는 다음과 같은 카드가 생성됐다.

```text
관련 이슈
OLED 패널 번인(Burn-in) 가능성
OLED 패널의 가장 큰 고질병 중 하나로, 정적인 요소가 장시간 표시될 경우 잔상이 발생할 수 있습니다.

관련 이슈
AS/RMA 관련 이전 및 해외 직구품 여부
Valve의 AS 정책은 구매자에게 귀속되는 경우가 많아 최초 구매자의 영수증이나 구매 내역이 있어야 AS를 받을 수 있습니다.

관련 이슈
SSD 불법 개조 및 교체 여부
일부 사용자가 저장 용량 확대를 위해 SSD를 임의로 교체하는 경우가 있으며, 교체 과정에서 다른 문제가 발생하지 않았는지 확인해야 합니다.

고질병 조사
아날로그 스틱 쏠림(Drift)
스틱을 건드리지 않아도 캐릭터나 커서가 움직이는 현상으로, SteamOS의 컨트롤러 테스트 기능으로 확인해야 합니다.

고질병 조사
USB-C 포트 마모 및 불량
충전 및 외부 기기 연결에 쓰이는 포트이므로, 잦은 사용으로 단자가 마모되거나 손상될 수 있습니다.
```

마지막으로 화면 표현도 조정했다. 처음에는 각 이슈 카드가 전체 가로폭을 차지해 내용에 비해 여백이 너무 컸다. 그래서 `.stage-two-risk-card`를 전체 가로형 카드에서 3열 미니블록 형태로 바꾸고, 세로 높이도 줄였다. 제품 정리 완료 후에는 `다음 단계로 넘어가기` 버튼 없이 2단계 분석이 자동 실행되도록 바꿨다.

추가 UX 정리:

- 제품 정리 카드에서 검색어 칩과 `비슷한 매물 찾기` 버튼은 숨기고, `다나와 검색`만 오른쪽 액션 영역에 표시한다.
- 제품 설명은 길 때만 내부 스크롤이 생기도록 하고, 테두리에 텍스트가 닿지 않게 패딩을 조정했다.
- 2단계 카드 등장/hover 애니메이션은 1단계 카드와 같은 느낌으로 통일했다.
- 이미지 새로고침은 제품 정리 카드만 갱신하고, 2단계 리스크 분석 상태를 다시 로딩으로 되돌리지 않게 분리했다.
- 최근 매물 drawer에서 과거 매물을 선택한 상태로 비교 동작을 해도 최신 매물로 튀지 않도록 선택 상태를 보존한다.

이 과정에서 얻은 결론은 다음과 같다.

- 모델이 못하는 것처럼 보일 때도, 실제 원인은 프롬프트의 출력 형식 강제일 수 있다.
- 검색·추론 단계와 JSON 구조화 단계를 나누면 품질이 좋아진다.
- 디버그용 직접 대화창은 모델 능력과 앱 파이프라인 문제를 분리해서 확인하는 데 유용하다.
- 제품군별 예시를 프롬프트에 직접 박으면 다른 제품에서 오판이 생긴다. 최종적으로는 “제품명은 이미 식별됐다고 보고, 제품 진위 검증이 아니라 중고 구매 리스크만 조사한다”는 역할 경계가 가장 안정적이었다.
- 발표에서는 이 사례를 "프롬프트를 세게 쓰는 것보다, 역할을 작게 나누고 역할 경계를 명확히 하는 것이 더 안정적이었다"는 개선 사례로 설명할 수 있다.

## 15. 제품 식별 — 세대·후속작 번호 보존 (2026-05-21)

**증상:** 매물 제목·사진에 `스위치2` / `Switch 2`가 분명한데 1단계 제품 정리가 `닌텐도 스위치` 또는 `스위치 OLED`로 일반화됨. 이후 2단계 리스크도 잘못된 제품 기준(조이콘 드리프트·OLED 번인 등)으로 채워짐.

**원인:** Gemini가 “알려진 공식 제품” 검색 결과에 끌려 후속작·비공식 명칭을 기존 세대/변형으로 접는 경향. 프롬프트에 “숫자·세대는 식별자”라는 규칙이 없었고, 서버 `cleanProductName()`도 제목의 `2`를 복원하지 않음.

**대응 (제품군 하드코딩 없이 절차형):**

1. `prompts/product-identify.txt`, `product-summary.txt` — 숫자·세대·후속작 번호·OLED/Pro/Plus/Max/Lite는 핵심 식별자이며 제거·일반화 금지.
2. `analyzer-server.mjs` — `preserveVariantTokens(productName, fallbackTitle)`  
   - 제목/본문에 `스위치2`·`Switch 2`가 있는데 출력이 `스위치`만이면 `2` 복원  
   - fallback에 `OLED` 등이 있는데 출력에 없으면 토큰 추가

**발표 한 줄:** “모델이 최신 공식 제품으로 끌려가는 것”은 프롬프트만이 아니라 **식별 결과 후처리**까지 같이 막아야 한다.

## 16. 매물 사진 분석 — 홍보용·스크랩 이미지 식별 (2026-05-21, 구현 반영)

2단계의 **판매자 이미지 분석**(`/api/listing-image-analysis`, 프롬프트는 `analyzer-server.mjs`의 `runListingImageAnalysis` 인라인 문자열)은 매물 사진마다 하자·구성품·박스 상태 코멘트를 만든다. 그런데 중고 판매글에 **판매자가 직접 찍은 실물 사진이 아니라**, 쇼핑몰·공식몰·검색에서 **긁어 온 홍보용 제품 이미지**만 올리는 경우가 많다.

### 증상 (실제 테스트)

- 닌텐도 스위치 2 마리오 카트 월드 세트 매물에서, 첫 장 사진이 **완성도 높은 공식 박스 아트·스튜디오 조명**처럼 보임.
- AI는 `제품 박스 상태` 배지와 함께 **「제품 박스 전면은 오염이나 파손 없이 깔끔해 보입니다」**처럼 답함.
- 화면에는 원가·할인가·`쿠폰보기` 같은 **상품 상세 UI 흔적**도 함께 보이는데, 이는 판매자 실물 촬영이 아니라 **다른 페이지에서 가져온 홍보 캡처**일 가능성이 큼.
- 결과적으로 **중고 실물 하자를 확인했다는 착각**이 생기고, 본문·2단계 리스크와 연결된 “직접 확인 필요” 메시지가 약해짐.

**발표용 캡처:** `발표용 이미지 캡처/대화첨부_42_image-b522681d-437c-4514-ab4b-c3ff34040a60.png`

### 왜 문제인가

- 홍보 이미지는 **제품 식별**에는 도움이 될 수 있지만, **실물 상태·하자·구성품·개봉 여부** 판단 근거가 되면 안 된다.
- 검색어 프롬프트에서 이미 “판매글 홍보 문구 제거”를 강조했듯, **사진 쪽에도 ‘판매글 맥락’과 ‘실물 근거’를 나누는 절차**가 필요하다.
- 제품 요약용 대표 이미지 검색(`scoreProductImageResult`에서 `official|공식|store` 가산)과 **매물 사진 하자 분석**은 목적이 다르다. 전자는 제품 확인, 후자는 **이 판매자가 올린 사진이 실물 근거인지**가 먼저다.

### 설계 방향 (프롬프트 보강안)

특정 쇼핑몰 URL이나 브랜드를 하드코딩하지 않고, **모든 매물에 적용되는 판별 절차**를 `runListingImageAnalysis` 규칙에 추가했다.

1. **먼저 사진 유형을 분류한다**
   - `seller_photo`: 판매자가 직접 찍은 실물·구성품·박스·하자 근거로 보이는 사진
   - `promo_or_scraped`: 공식 홍보샷, 쇼핑몰 상세 캡처, 스톡/렌더, 완성도 과도한 제품 단독 컷
   - `unclear`: 판별 어려움

2. **홍보용·스크랩으로 보이는 시각 단서 (복수면 가중)**
   - 배경이 지나치게 깨끗하고 조명·구도가 카탈로그/광고 수준
   - 제품만 크게 나오고 **책상·손·바닥·실사용 흔적**이 없음
   - 화면에 **가격·할인율·쿠폰·장바구니·리뷰 탭·쇼핑몰 UI**가 함께 보임
   - 박스/본체가 **완벽한 공식 패키지 아트**만 보이고 모서리 마모·개봉 흔적·스티커 제거 흔적이 없음
   - 본문에 “실사”, “직접 촬영”, “개봉”, “사용감” 언급이 있는데 사진은 광고컷과 불일치

3. **코멘트·배지·level 규칙**
   - `promo_or_scraped`이면 **실물 상태를 단정하지 않는다**. 예: `인터넷 홍보용 제품 이미지로 보여, 이 사진만으로 박스·하자·구성품 상태를 확인할 수 없습니다.`
   - `label` 예: `홍보 이미지`, `공식 홍보샷`, `실물 확인 불가`
   - 가격·할인율·쿠폰·장바구니·리뷰·상품명/가격 카드 같은 쇼핑몰 UI가 보이면 실제 상품이 함께 보여도 무조건 `promo_or_scraped`로 분류한다.
   - `level`은 보통 `caution` 또는 `neutral` (실제 하자가 보일 때만 `risk`)
   - `promo_or_scraped`에서는 `깨끗해 보입니다`, `양호합니다`, `오염이나 파손이 없습니다`, `본문 설명과 일치합니다`처럼 실물 상태를 확인한 듯한 표현을 금지한다.
   - `seller_photo`일 때만 찍힘·스크래치·구성품 누락·박스 상태를 구체적으로 적는다.

4. **`overall` 한 줄**
   - 홍보·스크랩 사진 비율이 높으면: `홍보용 이미지가 많아 실물 상태는 직거래·추가 실사진 확인이 필요합니다.`처럼 **사진 전체 신뢰도**를 요약한다.

### 발표에 넣을 문장

- “중고 매물 AI는 **제품이 뭔지**와 **지금 판매 중인 실물이 어떤지**를 구분해야 한다. 홍보 이미지를 실물 하자 없음으로 읽으면 오히려 위험하다.”
- “검색어에서 홍보 문구를 빼 듯, **사진 분석에서도 홍보 컷을 걸러내는 절차**를 프롬프트에 넣는다.”

### 구현 메모

- 코드 위치: `analyzer-server.mjs` → `runListingImageAnalysis` (현재 별도 `prompts/*.txt` 없음).
- `analyzer/app.js` → `imageAnalysisLabel()` fallback도 보강해, 모델이 라벨을 명시하지 않아도 코멘트에 홍보/공식/스크랩 단서가 있으면 `홍보 이미지` 또는 `실물 확인 불가` 배지가 뜨도록 했다.
- 이미지 분석 캐시는 새 프롬프트 확인을 위해 `ulsa_ai_analysis_cache_v10`으로 갱신했다.
- 추후 유지보수를 위해 `prompts/listing-image-analysis.txt` 분리도 검토 가능 (§14 고질병처럼 발표·수정 이력 관리용).
- UI는 기존 `image-analysis-badge`·`image-analysis-comment`를 그대로 쓰고, 프롬프트가 `홍보 이미지` 라벨을 내도록 맞췄다.

### 슬라이드 반영

§7 4번 “사진 하자 탐지”에 **「홍보 이미지 vs 실물 사진 구분」**을 한 줄 추가하면, 단순 비전 OCR이 아니라 **구매 안전을 위한 프롬프트 설계**로 설명하기 좋다.
